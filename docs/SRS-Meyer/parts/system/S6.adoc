=== (S.6) Verification and acceptance criteria

ifdef::env-draft[]
TIP: _Specification of the conditions under which an implementation will be deemed satisfactory. Here, "verification" as shorthand for what is more explicitly called "Verification & Validation" (V&V), covering several levels of testing — module testing, integration testing, system testing, user acceptance testing — as well as other techniques such as static analysis and, when applicable, program proving._  <<BM22>>
endif::[]

Verification ensures that {project-title} meets the requirements defined in <<s2>> and <<s3>>, while acceptance confirms the system fulfills user expectations. Testing will be conducted through the following methods:

* **Unit Testing** – verify that each core module (e.g., note management, canvas engine, geometry rendering) works as intended.
* **Integration Testing** – ensure that modules interact correctly, such as between the text editor and the file system.
* **System Testing** – validate end-to-end workflows like creating, editing, and saving notes.
* **User Acceptance Testing (UAT)** – confirm usability and responsiveness meet project expectations.

==== Acceptance Criteria

|===
| Category | Criteria | Reference

| Functional Requirements | All functionalities listed in <<s2>> execute correctly without errors or data loss under normal use. |
| Responsiveness | Canvas interactions (drag, resize, type) respond within 50ms; keyboard commands execute within 30ms of keypress. | <<NF212>>, <<NF221>>
| Rendering performance | The application maintains at least 60 fps for 95% of the time during continuous interaction on a canvas with up to 100 elements. | <<NF231>>
| Scalability | The application remains responsive with documents containing more than 500 on-canvas elements. | <<NF218>>
| Storage efficiency | A note with up to 1000 words and 10 shapes occupies less than 100KB on disk. | <<NF242>>
| Interfaces | The user interface is consistent with the design specification and supports keyboard-only navigation meeting WCAG standards. | <<NF213>>, <<NF217>>
| Reliability | Save and export operations are atomic; no data loss or file corruption occurs across sessions. | <<NF215>>, <<NF241>>
|===

The project will be considered accepted when all core functions operate without critical issues, all quantitative thresholds above are met, and usability testing confirms the system aligns with user expectations.
